# 剧本杀 Telegram Bot - PRD v2

**版本**: v2.1  
**日期**: 2026-03-27  
**状态**: 可评审草案

---

## 1. 产品简介（一句话讲清）

这是一个在 Telegram 群内运行的剧本杀 Bot，支持房主开局、玩家入房、自动发身份与线索、群内讨论和投票结算，目标是让 4-8 人在 20-40 分钟内完整跑完一局。

---

## 2. 完整用户流程（从入口到退出）

1. 用户私聊 Bot 执行 `/start` 完成注册，记录私聊通道。  
2. 房主在群里执行 `/create` 选择剧本，系统生成邀请码并创建房间。  
3. 玩家通过 `/join <邀请码>` 入房，房主用 `/room` 查看人数是否齐。  
4. 房主执行 `/start_game` 开局，Bot 校验人数与玩家私聊就绪状态。  
5. Bot 为每位玩家随机分配角色，并通过私聊发送角色剧本。  
6. Bot 紧接着私聊发送角色专属线索卡，群内仅提示“线索已发放”。  
7. 进入讨论阶段（默认 15 分钟，可房主提前结束）。  
8. Bot 在群内发起投票，每名玩家只能投一次，不可修改。  
9. 投票完成或超时后，Bot 揭晓真凶、公布结算结果并结束房间。  
10. 用户可用 `/history` 查看个人历史局结果，房主可重新开新局。

## 3. 每个功能的详细规则（不是“用户可以按要”，而是可执行规则）

### 3.1 房间与状态机

- 房间状态固定为: `WAITING -> RUNNING -> VOTING -> ENDED`。  
- 禁止越级切换状态；每次状态切换必须经过前置校验。  
- `WAITING` 阶段允许 `/join`、`/leave`，`RUNNING` 后禁止进出房。  
- 房主唯一，只有房主可执行 `/start_game` 与 `/end_discuss`。  
- 邀请码为 6 位大写字母数字，已结束房间的邀请码不可复用。

### 3.2 开局校验规则

- 人数必须等于剧本要求人数（如剧本要求 6 人，则少 1 人或多 1 人均不可开局）。  
- 所有玩家都必须先私聊 Bot 并执行过 `/start`。  
- 任意玩家未私聊就绪时，拒绝开局并在群内给出“谁未就绪 + 下一步操作”。  
- 同一房间只允许一次有效开局请求，重复点击需幂等返回“已开局”。

### 3.3 身份与线索发放规则

- 身份剧本与线索只能走私聊，群里不得出现可还原身份的信息。  
- 发放顺序固定: 先角色剧本，再线索卡。  
- 玩家可在游戏中用 `/script` 和 `/clues` 重看自己的内容。  
- 若私聊发送失败（如用户屏蔽 Bot），本局进入“阻塞态”，提示房主处理中断或踢人重开。

### 3.4 讨论与投票规则

- 讨论默认 15 分钟，房主可提前 `/end_discuss`。  
- 讨论结束后自动进入投票阶段。  
- 投票使用 `InlineKeyboard`，每位玩家仅可投一次，重复点击返回“你已投票”。  
- 全员投完立即结算；超时（默认 5 分钟）则按已投票数据结算。  
- 结算内容必须包含: 真凶身份、票型统计、每位玩家是否达成目标。

### 3.5 剧本上传规则

- 仅私聊可执行 `/upload_script`。  
- 上传流程为分步引导，必填项缺失不能提交。  
- 剧本结构必须包含: 基础信息、角色列表、真相。  
- 上传成功后默认可被所有房主选用（MVP 阶段无审核）。

### 3.6 指令权限矩阵（MVP）

| 指令 | 场景 | 权限 | 阶段限制 |
|---|---|---|---|
| `/start` | 私聊 | 全员 | 任意 |
| `/create` | 群聊 | 全员 | 无房间或上局结束 |
| `/join` | 群聊 | 全员 | `WAITING` |
| `/leave` | 群聊 | 非房主玩家 | `WAITING` |
| `/room` | 群聊 | 全员 | 任意 |
| `/start_game` | 群聊 | 房主 | `WAITING` |
| `/end_discuss` | 群聊 | 房主 | `RUNNING` |
| `/script` `/clues` | 私聊 | 房间内玩家 | `RUNNING`/`VOTING` |
| `/history` | 私聊 | 全员 | 任意 |

## 4. 边缘情况和错误状态（断网、重复提交、数据不存在怎么处理）

### 4.1 常见边缘场景

- 玩家在开局前退出群: 自动移出房间，人数不足则禁止开局。  
- 房主中途掉线: 房间不自动销毁，允许房主恢复后继续推进。  
- 投票消息被刷屏覆盖: 玩家仍可通过消息按钮投票，超时后自动结算。  
- Bot 重启: 从数据库恢复未结束局的状态与阶段。

### 4.2 错误处理规则

- **数据不存在**: 统一返回“房间不存在或已结束”。  
- **非法阶段操作**: 返回“当前阶段不可执行该命令 + 可执行命令提示”。  
- **重复请求**: 幂等处理，不重复扣状态、不重复结算。  
- **数据库写入失败**: 回滚事务，提示“系统繁忙，请重试”。  
- **第三方调用超时**: 快速失败并给出重试提示，不阻塞主流程。

### 4.3 错误码（内部约定）

- `ROOM_NOT_FOUND`  
- `PHASE_INVALID`  
- `PERMISSION_DENIED`  
- `PLAYER_NOT_READY_PM`  
- `DUPLICATE_VOTE`  
- `DB_WRITE_FAILED`

## 5. UI 设计（HTML demo 或 Figma 原型要能看清页面和数据来源）

### 5.1 页面清单（Mini App / Web 管理页预留）

1. 首页（剧本列表）  
2. 房间页（成员、状态、倒计时）  
3. 角色私密页（仅当前玩家可见）  
4. 投票页（候选角色）  
5. 结算页（真相与票型）

### 5.2 每个页面必须标注“数据从哪来”

- 首页剧本列表: **后端 API 返回**（`GET /api/scripts`）  
- 房间成员与状态: **后端 API 返回 + 本地倒计时计算**  
- 角色私密信息: **后端 API 返回（按用户鉴权）**  
- 投票候选列表: **后端 API 返回**  
- 用户昵称与头像: **Telegram WebApp SDK 读取**

### 5.3 原型完成标准

- 每个页面展示主状态、空状态、错误状态。  
- 每个交互按钮标注权限与触发条件。  
- 每个数据块标注来源（本地计算 / 后端 API / Telegram SDK）。

---

## 6. 非功能与验收标准

- 普通命令响应 < 1 秒。  
- 单实例支持同时 20 个房间进行中。  
- 任意异常不导致整局崩溃，至少可恢复或安全结束。  
- 验收口径: 连续跑通 10 局，无身份泄露、无重复结算、无卡死阶段。
